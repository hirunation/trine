# Support

Usage questions — how to compile, which lens to choose, how to interpret a similarity score — belong in GitHub Issues tagged `question`. Bug reports belong in Issues tagged `bug` with a minimal reproduction case, expected behavior, and actual behavior. Feature requests belong in Issues tagged `enhancement` with a concrete use case, not an abstract wish.

Architectural discussion — changes to the embedding algebra, new chain structures, modifications to the routing layer — belongs in pull requests with code, not in issues with prose. If you want to propose a structural change, build a prototype first. The system's design is best argued through working alternatives.

This boundary exists to prevent noise from overwhelming core development. Maintainer time is finite. Questions that are answered in the README, the BENCHMARK_REPORT, or the API headers will be closed with a link. Questions that reveal gaps in documentation will result in documentation improvements.
